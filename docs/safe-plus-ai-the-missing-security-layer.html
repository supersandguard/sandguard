Safe + AI: The Missing Security Layer Between Proposal and Signing

July 25, 2025 AI Agents Safe Security

AI agents are managing millions in crypto right now ‚Äî trading, treasury operations, DeFi strategies. But here's the question nobody's asking: who verifies what the agent is actually signing?

The Age of Autonomous Wallets

Something fundamental is shifting in how crypto wallets are managed. AI agents ‚Äî autonomous software systems that can reason, plan, and execute ‚Äî are increasingly taking on roles that used to belong exclusively to humans. Treasury management, yield optimization, automated rebalancing, cross-chain bridging. The list grows every week.

The numbers behind this shift are hard to ignore. Protocols like Gnosis's Olas have deployed thousands of autonomous agents managing onchain operations. Frameworks like ElizaOS and Virtuals are making it easier than ever to spin up agents that interact with DeFi protocols. Investment DAOs are experimenting with AI portfolio managers. And the capital flowing through agent-controlled addresses is measured in hundreds of millions.

Most of these agents interact with the blockchain through Safe multisig wallets ‚Äî the de facto standard for onchain security. And for good reason: Safe is battle-tested, open-source, and secures over $100 billion in assets. Every serious DAO, protocol treasury, and institutional fund runs on Safe.

But as AI agents move from passive tools to active proposers and signers, a critical gap has emerged in the security model. And it's one that the Safe ecosystem hasn't addressed yet.

The Verification Gap

Here's how the typical AI-managed Safe workflow looks today:

AI Agent proposes tx ‚Üí Raw calldata appears in queue ‚Üí Human signer reviews ‚Üí Signs (or doesn't)

That arrow between "raw calldata appears" and "human signer reviews" is where things fall apart. Because what exactly is the human reviewing?

In practice, the signer sees a transaction hash, maybe a destination address, and a blob of hexadecimal calldata. If they're lucky, the Safe UI can partially decode the function name. If the transaction involves batched calls, proxy interactions, or nested delegatecall patterns ‚Äî which is increasingly common for AI-generated transactions ‚Äî the signer sees effectively nothing useful.

So what happens? The signer falls back on trust. They trust that the AI agent proposed the right thing. They trust that the interface isn't lying to them. They trust that the calldata matches what they expect.

We've seen where that trust leads. The ByBit hack ‚Äî $1.43 billion stolen ‚Äî happened precisely because signers trusted what they saw instead of verifying what they were signing. That attack involved a human proposer, not an AI. Now imagine the same vulnerability with autonomous agents proposing dozens of transactions per day, each more complex than the last.

The uncomfortable truth: Multisig provides consensus ‚Äî multiple parties must agree. But consensus is meaningless if none of the agreeing parties can independently verify what they're agreeing to. Between "agent proposes" and "signer approves," there is currently zero verification. No decode. No simulation. Just blind trust.

Why This Gap Exists

This isn't a criticism of Safe. Safe built the gold standard for onchain authorization ‚Äî the idea that no single key should control significant assets. That was the right problem to solve first, and they solved it exceptionally well.

But the security model was designed for a world where humans propose and humans sign. In that world, the proposer can explain the transaction over a call. The signer can ask questions. Context flows through human channels ‚Äî Telegram groups, governance forums, team meetings.

AI agents break that model. An agent doesn't hop on a call to explain its reasoning. It doesn't write a proposal summary in your Discord. It generates calldata and submits it to the queue. The transaction is the communication ‚Äî and if you can't read the transaction, you're flying blind.

Safe's module system (ERC-7579) has started addressing adjacent concerns. Recovery modules handle key loss. Dead man's switches prevent lockout. Spending limits cap exposure. These are important. But they're all policy controls ‚Äî rules about what can happen. What's missing is a verification control ‚Äî understanding of what is happening, right now, in this specific transaction.

Verification as a Security Layer

The gap between proposal and signing is exactly where SandGuard sits. We built it because we kept seeing the same pattern: sophisticated onchain operations, competent signers, and a critical moment where everyone just... hopes for the best.

SandGuard is a transaction verification layer for Safe multisig. Before any signer approves, it performs three independent checks:

1. Calldata Decoding

Every transaction is decoded against verified ABIs into human-readable operations. Not just the top-level function ‚Äî the full call tree, including nested calls, proxy interactions, and batched operations. An AI agent proposing a complex DeFi strategy gets unpacked into plain English: "Swap 50 ETH for USDC on Uniswap V3, deposit USDC into Aave V3 as collateral, borrow 30,000 DAI against it."

2. Transaction Simulation

The transaction is executed against a fork of the current chain state. You see every balance change, every state modification, every approval and transfer ‚Äî before anyone signs. If an agent proposes something that would drain the treasury or grant unlimited token approvals to an unverified contract, the simulation makes it impossible to miss.

3. Risk Scoring

An automated risk engine evaluates the transaction against known attack patterns and contextual red flags. Interactions with unverified contracts, unusual value transfers, proxy upgrades, first-time counterparties ‚Äî each factor contributes to a composite risk score that gives signers an immediate read on whether something deserves extra scrutiny.

The key principle: Verification is independent of the proposer. It doesn't matter whether a transaction was proposed by a trusted teammate, an AI agent, or a compromised interface. The calldata is the calldata. Decode it, simulate it, score it ‚Äî every time, automatically.

Agents Verifying Agents

There's a deeper architectural point here that's worth dwelling on. As AI agents become more capable and more autonomous, the security model needs to evolve beyond "human reviews everything." Humans are already struggling to verify complex DeFi transactions proposed by other humans. Adding AI-generated complexity only makes it worse.

The natural evolution is agents verifying agents. An AI verification layer that can decode, simulate, and assess risk at machine speed ‚Äî serving as a check on the proposing agent before any human is even asked to sign. Think of it as separation of concerns applied to AI: the agent that proposes should never be the same system that verifies.

This is how traditional finance works. The trader who places the order isn't the same person who settles it. The auditor is independent of the team being audited. In onchain security, we've been asking the same entity (or the same compromised UI) to both propose and validate. That's not a security model. It's a trust exercise.

An Ecosystem Problem

We're building SandGuard because this gap exists today and assets are at risk now. But we don't think verification should be a single product from a single team. It should be an ecosystem layer ‚Äî as fundamental to the Safe stack as multisig itself.

The equation is straightforward:

Safe (consensus) + Verification (understanding) = Actual security

Consensus without understanding is ceremony. Understanding without consensus is centralization. You need both. And as AI agents take on more responsibility onchain, the verification layer becomes not just useful but essential infrastructure.

Safe's module architecture is designed for exactly this kind of extensibility. A verification module ‚Äî whether it's SandGuard or something else ‚Äî could plug directly into the signing flow, making decoded analysis and simulation a mandatory step before execution. Not an optional extra. Not a "nice to have." A core part of the security model.

The space is wide open. The stakes ‚Äî billions in AI-managed assets with no verification layer ‚Äî are too high to build in silos. If you're working on AI agents that interact with Safe multisigs, or thinking about onchain AI security more broadly, this is the conversation worth having.
